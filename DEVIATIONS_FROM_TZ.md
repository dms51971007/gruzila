## Отклонения от ТЗ для `gruzilla-executor`

Этот файл фиксирует все известные отличия текущей реализации от исходного ТЗ по компоненту `gruzilla-executor` и сопутствующим частям.

### 1. Поддерживаемые типы шагов сценария

- **Соответствие ТЗ**
  - Поддерживаются шаги типов `rest` и `kafka` (как и описано в ТЗ).
- **Расширение ТЗ**
  - Добавлен тип шага `db` для проверки записей в PostgreSQL (поля `db_dsn`, `db_query`, `assert.rows`).
- **Отклонение от ТЗ**
  - Тип шага `mq` **формально разрешён в YAML** (валидатор принимает `type: mq`), но в рантайме:
    - валидация не накладывает требований к полям MQ;
    - исполнение `mq` пока **не реализовано**: при выполнении шага возвращается ошибка `"mq step is not implemented yet"`.

### 2. Метрики и их API

- **Что было в ТЗ**
  - Требовались: время работы, количество успешных/неуспешных итераций, текущий TPS и «прочие агрегаты».
  - Отдача через `POST /api/v1/metrics` в формате `{ "status", "data" }`.
- **Как сделано сейчас**
  - Структура `Metrics`:
    - `attempts_count` — всего запущено попыток сценария (success + error + in-flight).
    - `success_count` — количество успешно завершённых сценариев.
    - `error_count` — количество завершившихся ошибкой сценариев (включая паники в шагах).
    - `last_latency_ms` — латентность последнего завершившегося запуска сценария.
    - `target_tps` — целевой TPS в текущую секунду (результат функции `effectiveTPS` после учёта процента и ramp-up).
    - `current_tps` — **реальный TPS**: прирост `attempts_count` за последнюю секунду (дельта между тиками), а не «запланированное» число итераций.
  - Метрики отдаются:
    - через `POST /api/v1/metrics` — только `Metrics`;
    - дублируются внутри `Status` в ответах `start/stop/update/status/reload`.
- **Отклонения**
  - В явном виде **нет поля «время работы»**, вместо него:
    - в `Status` есть `started_at` (время старта нагрузки), время работы получается как `now - started_at` на стороне клиента.
  - Появились дополнительные поля (`attempts_count`, `target_tps`), которых не было в ТЗ — это **расширение** для диагностики нагрузки.

### 3. Модель нагрузки и ramp-up

- **ТЗ**
  - В сценарии задаётся **базовый TPS** и **скорость набора TPS** (пример из ТЗ: `10 TPS/min`).
  - CLI передаёт коэффициент (1–500%) и, при необходимости, уточнённый TPS.
- **Реализация**
  - В `RunConfig`:
    - `percent` — коэффициент нагрузки (если ≤ 0, подставляется 100);
    - `base_tps` — базовый TPS сценария (если ≤ 0, подставляется 1);
    - `ramp_up_seconds` — длительность **линейного** набора нагрузки от 0 до `base_tps * percent/100`.
  - Эффективный TPS считается функцией:
    - `effectiveTPS(base_tps, percent, ramp_up_seconds, started_at)`.
- **Отклонения**
  - Вместо «скорости набора TPS в TPS/минуту» используется **время разгона в секундах** (`ramp_up_seconds`), с **линейной** функцией роста.
  - На серверной стороне **нет жёсткой валидации** диапазона `percent` (ТЗ говорило 1–500%). Значения за пределами этого диапазона технически возможны, объектно не блокируются.

### 4. HTTP API executor'а

- **Соответствие ТЗ**
  - Все основные эндпоинты реализованы:
    - `POST /api/v1/start`
    - `POST /api/v1/stop`
    - `POST /api/v1/update`
    - `POST /api/v1/status`
    - `POST /api/v1/metrics`
  - Формат ответа соответствует стандарту REST API проекта:
    - `{ "status": "success" | "error", "data"?: ..., "error"?: "..." }`.
  - Префикс `/api/v1/` и метод `POST` соблюдены.
- **Расширения ТЗ**
  - Добавлен эндпоинт `POST /api/v1/reload`:
    - перечитывает YAML-сценарий с диска без остановки нагрузки;
    - обновляет имя сценария и очищает `last_error`.
  - Добавлен эндпоинт `POST /api/v1/reset_metrics`:
    - обнуляет счётчики (`attempts_count`, `success_count`, `error_count`), `last_latency_ms`, `last_error`, `started_at`; разрешён только при остановленной нагрузке (`running: false`).
  - Добавлен эндпоинт `POST /api/v1/shutdown`:
    - инициирует корректное завершение процесса executor (вызов `http.Server.Shutdown`), чтобы CLI мог перезапустить процесс с новым кодом.
  - Добавлен эндпоинт **GET /metrics** (не POST, без авторизации):
    - отдача метрик в формате Prometheus для сбора сборщиком (Prometheus) и визуализации в Grafana.
  - В `Status` дополнительно отдаётся:
    - `scenario_path` — путь до файла сценария;
    - `started_at` — момент старта нагрузки.

### 5. Поведение при ошибках и паниках

- **ТЗ (ожидаемое поведение)**
  - Явного описания паник нет, подразумевалось, что ошибки шагов считаются в `error_count`.
- **Реализация**
  - Внутри выполнения сценария добавлен `recover()`:
    - паника в любом шаге оборачивается в ошибку вида `panic: ...`;
    - такое выполнение учитывается в `error_count` и попадает в `last_error`.
- **Отклонения**
  - Это поведение является **уточнением/расширением** ТЗ: паники явно трактуются как ошибки сценария и учитываются в метриках, чтобы не «терять» попытки при высокой нагрузке.

### 6. Счётчики под высокой нагрузкой

- **ТЗ**
  - Требовались корректные счётчики успешных/неуспешных итераций и текущий TPS.
- **Реализация**
  - `success_count`, `error_count`, `attempts_count`, `last_latency_ms` реализованы на **атомарных** счётчиках (`sync/atomic`), чтобы:
    - избежать потери инкрементов при большом числе параллельных горутин;
    - корректно подсчитывать общее количество попыток.
- **Отклонения**
  - Это техническое отличие реализации (ТЗ не регламентировало механизм), по функционалу соответствует/расширяет требования.

### 7. Шаблоны и переменные

- **ТЗ**
  - Описывалось использование шаблонов из папки `templates/` и подстановки переменных `{{var}}` в шагах.
- **Реализация**
  - Шаблоны для тел сообщений реализованы через отдельный пакет `internal/templates` и поле `template` в шаге:
    - если `template` задан, тело шага рендерится через Go `text/template` с передачей всех переменных;
    - если нет — используется поле `body` с простой подстановкой `{{var}}`.
- **Отклонения**
  - Существенных функциональных отклонений от ТЗ нет; реализация местами богаче (кэширование шаблонов, пул буферов), но это детали реализации.

### 8. Prometheus и мониторинг

- **ТЗ**
  - Не описывало интеграцию со сборщиком метрик или Grafana.
- **Реализация**
  - В executor добавлена отдача метрик Prometheus: **GET /metrics** (формат Prometheus text).
  - Метрики (namespace `gruzilla_`, лейбл `scenario`): `attempts_total`, `success_total`, `errors_total`, `current_tps`, `target_tps`, `last_latency_ms`, `running`.
  - Обновление: каждый тик в runLoop и при Stop(); значения синхронизированы с атомарными счётчиками и статусом.
  - В репозитории добавлены `docker-compose.monitoring.yml` и `monitoring/prometheus.yml` для запуска Prometheus и Grafana; в Grafana подключается datasource Prometheus (`http://prometheus:9090`), для «в секунду» используются запросы вида `rate(gruzilla_success_total[1m])`.
- **Отклонения**
  - Расширение ТЗ: полноценный стек мониторинга (Prometheus + Grafana) не был обязателен по ТЗ.

### 9. CLI: перезапуск executor и сброс метрик

- **ТЗ**
  - CLI управляет нагрузкой через API executor'а (start, stop, update, status); отдельно не регламентировались «перезапуск процесса» и «сброс счётчиков».
- **Реализация**
  - **`run reset-metrics`** — вызов `POST /api/v1/reset_metrics` для обнуления счётчиков при остановленной нагрузке.
  - **`executors restart`** — отправка `POST /api/v1/shutdown` текущему процессу, ожидание ~2 с, запуск нового процесса executor с теми же `--scenario` и `--addr` (чтобы подхватить обновлённые исходники).
- **Отклонения**
  - Расширение ТЗ: удобство эксплуатации (перезапуск без ручного поиска PID, сброс метрик без повторного start/stop).
